Scope used for this document
----------------------------
This is the strict scope:

1. All live VULCAN-master functionality should exist in the JAX tree.
2. All numerical / physics / preprocessing paths that affect a run should be
   implemented in JAX rather than vendored NumPy / SciPy code.
3. The JAX tree should not need VULCAN-master at runtime or for validation of
   supported configs.
4. Utility / plotting / diagnostic surface should be present in the JAX repo
   as well, so the repo is functionally complete rather than only
   "solver-complete".

Dependency policy for this document
-----------------------------------
External dependencies that VULCAN-master already calls do NOT need to be
rewritten. The goal is to replace VULCAN itself, not every third-party tool it
uses.

Examples:
- FastChem can remain an external solver, because VULCAN-master already shells
  out to it for EQ initialization.
- matplotlib / PIL do not need JAX rewrites.
- NumPy / SciPy used inside VULCAN-owned code are still fair rewrite targets,
  because that is part of replacing VULCAN's implementation.

Important scope note
--------------------
Ros2 is the only live solver in VULCAN-master. The other solver classes in
op.py are commented-out dead code, not active user-facing functionality.
Therefore there is no real missing non-Ros2 solver port unless the goal is to
resurrect dead code as a separate project.

These are still remaining tasks, but they are already owned elsewhere:

- Phase 16 suite-ordering regression
- Phase 17 compensated chem_rhs summation
- Phase 18 per-test fixture sharing

Current high-level status
-------------------------
The core Ros2 runtime is mostly rewritten in JAX already:

- JAX chemistry RHS / Jacobian
- JAX diffusion + block solves
- JAX Ros2 step
- JAX outer loop
- JAX photo kernels
- JAX condensation kernels
- JAX atm-refresh kernels

The largest remaining gap is that the repo is not yet a complete JAX rewrite

- rate / cross-section / photo-bin preprocessing
- EQ initialization wrapper logic around the external FastChem solver
- repo-surface parity (diagnostics, tools, plot scripts, remaining helper/docs)
- validation breadth across the full config matrix

As of 2026-04-27, the non-code runtime data files under `atm/` and `thermo/`
that the current JAX code can select are vendored in `VULCAN-JAX/`. The
remaining tree deltas in those areas are helper scripts / README files only,
plus the non-runtime FastChem source/build tree.

D2. Compensated chem_rhs summation  [WONTFIX, maintainer policy]
---------------------------------
Status:
- WONTFIX. The user has explicitly opted to leave the documented chem_rhs
  cancellation floor in place. Do not attempt to close it.

What it is:
- chem_rhs has a known, documented ~1e-4 relerr vs VULCAN-master for a small
  set of species (CH2_1, HC3N, HCCO at certain layers). Phase 17 traced this
  to per-term ulp drift between JAX's `jnp.prod(y**stoich)` and master's
  SymPy-emitted `y[A] * y[B] * y[C]` -- not a summation-order issue
  (Neumaier-compensated summation does NOT close the gap).
- Closing the gap requires emitting each term in master's SymPy order via
  codegen. Stage 3 of the original Phase 17 plan. Pays a ~14x per-call cost
  on the per-step hot path; the trade is net-negative for the project.

Why this is permanent:
- The user has decided this floor is acceptable. Future agents:
  - Document the floor when comparing JAX outputs to master.
  - Do not work on Stage 3 codegen.
  - Do not relax test tolerances to hide the floor (current rtol=1e-3 on
    chem_rhs is the right tolerance; the Jacobian, which has less per-entry
    cancellation, agrees to machine precision).
- See CLAUDE.md "Numerical hygiene -> chem_rhs cancellation floor" and
  README.md "Validation" for the canonical write-up.

Effort:
- N/A (WONTFIX).
